# Requirements Document

## Introduction

Данная функциональность добавляет Telegram бот интерфейс к существующей системе генерации видео (TikTok-стиль). Бот позволяет пользователям взаимодействовать с пайплайном генерации видео через Telegram: отправлять описание желаемого видео, выбирать параметры (длительность, соотношение сторон, голос, стиль), получать сгенерированный видеоконтент прямо в чате. Бот работает от имени зарегистрированного Telegram-бота и обрабатывает запросы асинхронно, уведомляя пользователя о прогрессе генерации.

## Glossary

- **Bot**: Telegram-бот, зарегистрированный через BotFather, через который пользователи взаимодействуют с системой.
- **User**: Telegram-пользователь, отправляющий запросы боту.
- **Generation_Request**: Запрос пользователя на создание видео, содержащий описание и параметры.
- **Pipeline**: Существующий workflow генерации видео (`VideoGenerationWorkflow`), включающий генерацию текста, изображений, видео и аудио.
- **Session**: Изолированный контекст выполнения одного запроса на генерацию видео, хранящий промежуточные результаты.
- **Progress_Notification**: Сообщение бота пользователю об этапе выполнения генерации.
- **Telegram_API**: Официальный Bot API Telegram для отправки и получения сообщений.
- **Webhook**: HTTP-эндпоинт, на который Telegram отправляет обновления (входящие сообщения).
- **Polling**: Режим получения обновлений от Telegram путём периодических запросов к API (альтернатива Webhook).
- **Inline_Keyboard**: Интерактивные кнопки в сообщениях Telegram для выбора параметров.
- **Aspect_Ratio**: Соотношение сторон видео: `9:16` (вертикальное, TikTok-стиль) или `16:9` (горизонтальное).
- **Voice_ID**: Идентификатор голоса ElevenLabs для озвучки нарратива.
- **Style_Prompt**: Текстовое описание визуального стиля, применяемого к генерации изображений.
- **Free_Mode**: Режим использования бесплатных AI-моделей вместо платных.

## Requirements

### Requirement 1: Инициализация и запуск бота

**User Story:** Как администратор системы, я хочу запускать Telegram бота как отдельный процесс, чтобы бот был доступен пользователям независимо от CLI-интерфейса.

#### Acceptance Criteria

1. THE Bot SHALL читать токен Telegram Bot API из переменной окружения `TELEGRAM_BOT_TOKEN` при запуске.
2. IF переменная окружения `TELEGRAM_BOT_TOKEN` не задана или пуста, THEN THE Bot SHALL завершить процесс с кодом ошибки и вывести сообщение об ошибке в stderr.
3. WHEN Bot запускается, THE Bot SHALL установить соединение с Telegram API в режиме long polling.
4. WHEN Bot успешно подключается к Telegram API, THE Bot SHALL вывести в stdout имя бота и его username.
5. THE Bot SHALL обрабатывать сигналы завершения процесса (SIGINT, SIGTERM) и корректно завершать работу, останавливая polling.

---

### Requirement 2: Команда /start и приветственное сообщение

**User Story:** Как пользователь Telegram, я хочу получить приветственное сообщение при первом обращении к боту, чтобы понять, как им пользоваться.

#### Acceptance Criteria

1. WHEN User отправляет команду `/start`, THE Bot SHALL ответить приветственным сообщением с описанием возможностей и инструкцией по использованию.
2. WHEN User отправляет команду `/help`, THE Bot SHALL ответить сообщением с перечнем доступных команд и их описанием.
3. THE Bot SHALL включать в приветственное сообщение инструкцию о том, что для начала генерации нужно отправить текстовое описание видео.

---

### Requirement 3: Приём запроса на генерацию видео

**User Story:** Как пользователь Telegram, я хочу отправить текстовое описание видео боту, чтобы запустить процесс генерации.

#### Acceptance Criteria

1. WHEN User отправляет текстовое сообщение (не команду), THE Bot SHALL интерпретировать его как описание для генерации видео и начать диалог выбора параметров.
2. THE Bot SHALL принимать описания на русском и английском языках.
3. IF User уже имеет активную Generation_Request в процессе выполнения, THEN THE Bot SHALL уведомить User о том, что предыдущая генерация ещё не завершена, и предложить дождаться её завершения.
4. WHEN Bot получает описание от User, THE Bot SHALL подтвердить получение сообщением с кратким резюме описания.

---

### Requirement 4: Выбор параметров через Inline Keyboard

**User Story:** Как пользователь Telegram, я хочу выбирать параметры генерации видео через кнопки в чате, чтобы не вводить их вручную.

#### Acceptance Criteria

1. WHEN Bot получает описание видео, THE Bot SHALL отправить сообщение с Inline_Keyboard для выбора длительности видео (15, 30, 60 секунд).
2. WHEN User выбирает длительность, THE Bot SHALL отправить сообщение с Inline_Keyboard для выбора Aspect_Ratio (`9:16` — вертикальное, `16:9` — горизонтальное).
3. WHEN User выбирает Aspect_Ratio, THE Bot SHALL отправить сообщение с Inline_Keyboard для выбора голоса из доступных голосов ElevenLabs (минимум 3 варианта с именами).
4. WHEN User выбирает голос, THE Bot SHALL отправить сообщение с Inline_Keyboard для выбора режима моделей (стандартный или Free_Mode).
5. WHEN User завершает выбор всех параметров, THE Bot SHALL отобразить сводку выбранных параметров и запросить подтверждение через Inline_Keyboard (кнопки «Начать генерацию» и «Отмена»).
6. IF User нажимает «Отмена», THEN THE Bot SHALL отменить Generation_Request и предложить начать заново.

---

### Requirement 5: Асинхронная генерация с уведомлениями о прогрессе

**User Story:** Как пользователь Telegram, я хочу получать уведомления о ходе генерации видео, чтобы знать, на каком этапе находится процесс.

#### Acceptance Criteria

1. WHEN User подтверждает запуск генерации, THE Bot SHALL запустить Pipeline асинхронно и немедленно ответить сообщением о начале генерации.
2. WHILE Pipeline выполняется, THE Bot SHALL отправлять Progress_Notification при переходе между этапами: генерация текста истории, генерация промптов, генерация изображений, генерация видео-сегментов, генерация аудио, сборка финального видео.
3. WHEN Pipeline генерирует видео-сегменты, THE Bot SHALL отправлять Progress_Notification с указанием количества завершённых сегментов (например, «Видео: 3/15 сегментов»).
4. THE Bot SHALL обновлять существующее сообщение о прогрессе вместо отправки новых сообщений для каждого обновления этапа.
5. IF Pipeline завершается с ошибкой, THEN THE Bot SHALL отправить User сообщение об ошибке с кратким описанием причины и предложением повторить попытку.

---

### Requirement 6: Доставка готового видео

**User Story:** Как пользователь Telegram, я хочу получить готовое видео прямо в Telegram-чате, чтобы сразу его просмотреть и поделиться.

#### Acceptance Criteria

1. WHEN Pipeline успешно завершает генерацию финального видео, THE Bot SHALL отправить видеофайл User через Telegram API методом `sendVideo`.
2. THE Bot SHALL включать в сообщение с видео подпись с кратким описанием (исходный запрос пользователя, длительность, соотношение сторон).
3. IF размер финального видеофайла превышает 50 МБ (ограничение Telegram Bot API), THEN THE Bot SHALL отправить User уведомление о превышении лимита и предложить альтернативу (например, уменьшить длительность).
4. WHEN видео успешно отправлено, THE Bot SHALL отправить сообщение с предложением создать новое видео.

---

### Requirement 7: Команда /cancel — отмена активной генерации

**User Story:** Как пользователь Telegram, я хочу иметь возможность отменить текущую генерацию, чтобы не ждать завершения нежелательного запроса.

#### Acceptance Criteria

1. WHEN User отправляет команду `/cancel`, THE Bot SHALL проверить наличие активной Generation_Request для данного User.
2. IF активная Generation_Request существует, THEN THE Bot SHALL остановить выполнение Pipeline и уведомить User об успешной отмене.
3. IF активная Generation_Request отсутствует, THEN THE Bot SHALL уведомить User о том, что нет активных запросов для отмены.

---

### Requirement 8: Хранение состояния пользовательских сессий

**User Story:** Как система, я хочу хранить состояние диалога каждого пользователя, чтобы корректно обрабатывать многошаговый процесс выбора параметров.

#### Acceptance Criteria

1. THE Bot SHALL хранить состояние диалога (текущий шаг, выбранные параметры) для каждого User в памяти процесса, используя Telegram User ID в качестве ключа.
2. WHEN User завершает Generation_Request (успешно или с ошибкой), THE Bot SHALL очищать состояние диалога для данного User.
3. IF Bot перезапускается, THE Bot SHALL инициализировать пустое состояние для всех пользователей (состояние не персистируется между перезапусками).
4. THE Bot SHALL поддерживать одновременные Generation_Request от разных пользователей без взаимного влияния их состояний.

---

### Requirement 9: Обработка ошибок и устойчивость

**User Story:** Как пользователь Telegram, я хочу получать понятные сообщения об ошибках, чтобы знать, что пошло не так и что делать дальше.

#### Acceptance Criteria

1. IF Telegram API возвращает ошибку при отправке сообщения, THEN THE Bot SHALL повторить попытку отправки до 3 раз с экспоненциальной задержкой.
2. IF Pipeline завершается с необработанным исключением, THEN THE Bot SHALL перехватить исключение, записать его в лог и отправить User сообщение об ошибке.
3. WHEN Bot получает неизвестную команду (начинающуюся с `/`), THE Bot SHALL ответить сообщением о том, что команда не распознана, и предложить использовать `/help`.
4. THE Bot SHALL логировать все входящие сообщения, запуски генерации и ошибки в stdout с временными метками.

---

### Requirement 10: Конфигурация и переменные окружения

**User Story:** Как администратор системы, я хочу настраивать параметры бота через переменные окружения, чтобы не изменять исходный код при развёртывании.

#### Acceptance Criteria

1. THE Bot SHALL читать следующие переменные окружения: `TELEGRAM_BOT_TOKEN` (обязательная), `TELEGRAM_ALLOWED_USER_IDS` (опциональная, список разрешённых User ID через запятую).
2. WHERE `TELEGRAM_ALLOWED_USER_IDS` задана, THE Bot SHALL отвечать только пользователям, чьи Telegram User ID входят в список разрешённых.
3. IF User не входит в список разрешённых пользователей, THEN THE Bot SHALL ответить сообщением об ограниченном доступе и не обрабатывать запрос.
4. THE Bot SHALL использовать все существующие переменные окружения проекта (`FAL_API_KEY`, `ELEVENLABS_API_KEY`, `OPENROUTER_API_KEY` и др.) без дублирования конфигурации.
